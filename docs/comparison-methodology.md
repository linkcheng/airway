# Airway 分支对比测试方法论

> 同一需求（Clawith-Bisheng MCP 代理），四种开发方法论，对照实验。

## 一、实验设计

| 目录 | 方法 | 说明 |
|------|------|------|
| `main/` | Vibe Coding（无插件） | 基准对照组 |
| `superpower/` | Superpowers 插件 | 结构化流程（brainstorm → plan → TDD → review） |
| `openspec/` | OpenSpec 插件 | 变更提案驱动（propose → design → spec → implement） |
| `speckit/` | SpecKit 插件 | 规格驱动（specify → plan → tasks → implement） |

### 共同输入

来自 `CLAUDE.md` 的统一需求：
- 产品定位：MCP 代理，桥接 Clawith Agent 与 Bisheng RAG
- 核心约束：不修改上游源码、面向接口编程、无状态设计
- 本地参考：Bisheng 源码 + Clawith 源码

## 二、评估维度与评分标准

### A. Spec 质量（输入端）— 权重 30%

| 指标 | 5 分 | 3 分 | 1 分 |
|------|------|------|------|
| **完整性** | 覆盖所有 MVP 功能点+边界条件 | 覆盖核心功能，遗漏边界 | 功能点有明显缺失 |
| **精确性** | 无歧义，可直接指导实现 | 基本清晰，少量需澄清 | 存在歧义或矛盾 |
| **结构化** | 模板化组织，层次分明 | 有结构但不统一 | 自由格式，难定位 |
| **可追溯性** | 需求→设计→任务 链路完整 | 部分可追溯 | 无关联 |

### B. 代码质量（输出端）— 权重 40%

#### B1. 能不能跑（客观验证）

| 检查项 | 验证方法 | Pass/Fail |
|--------|----------|-----------|
| 依赖安装 | `uv sync` | 无报错 |
| 单元测试 | `uv run pytest tests/ -v` | 全部通过 |
| 代码检查 | `uv run ruff check .` | 无错误 |
| 类型检查 | `uv run mypy src/`（如有配置） | 无错误 |

#### B2. 架构合理性

| 指标 | 5 分 | 3 分 | 1 分 |
|------|------|------|------|
| **模块划分** | 单一职责，边界清晰 | 基本合理，有耦合 | 职责混乱 |
| **代码精简度** | 用最少代码实现需求 | 适度 | 过度设计或冗余 |
| **可扩展性** | 新增适配器/工具不需改核心 | 需少量改动 | 大面积修改 |
| **错误处理** | 完整的异常链+有意义的错误信息 | 基本覆盖 | 大面积裸 except |

#### B3. 量化指标

| 指标 | 收集方法 |
|------|----------|
| 源码行数（不含空行注释） | `find . -name "*.py" \| xargs wc -l` |
| 文件数量 | `find . -name "*.py" \| wc -l` |
| 依赖数量 | `cat pyproject.toml` |
| 测试行数 | `find tests/ -name "*.py" \| xargs wc -l` |
| 测试/代码比 | 测试行数 / 源码行数 |

### C. 方法论效率（过程端）— 权重 30%

| 指标 | 5 分 | 3 分 | 1 分 |
|------|------|------|------|
| **交互轮次** | ≤ 5 轮完成 | 6-10 轮 | > 10 轮 |
| **人工介入** | 零纠正 | 1-2 次纠正 | 多次纠正 |
| **一次性完成率** | 代码无需返工 | 小幅返工 | 重大返工 |
| **文档产出效率** | spec→code 比值合理 | 文档略多/少 | 文档远超代码量 |

## 三、执行计划

### Phase 1: 客观验证（自动化）

对每个分支依次执行：

```bash
cd <branch-dir>

# 1. 依赖安装
uv sync 2>&1 | tail -5

# 2. 单元测试
uv run pytest tests/ -v --tb=short 2>&1

# 3. 代码检查
uv run ruff check . 2>&1

# 4. 代码量统计
echo "=== 源码 ==="
find . -path ./.venv -prune -o -path ./uv.lock -prune -o -name "*.py" -print | xargs grep -cv '^\s*$\|^\s*#\|^\s*"""' 2>/dev/null || echo "N/A"

echo "=== 测试 ==="
find tests/ -name "*.py" | xargs grep -cv '^\s*$\|^\s*#\|^\s*"""' 2>/dev/null || echo "N/A"
```

### Phase 2: Spec 交叉对比（人工评审）

逐维度对四份 spec 打分，每个维度给出依据。

### Phase 3: 代码交叉对比（人工评审）

基于 Phase 1 数据 + 代码阅读，逐维度打分。

### Phase 4: 过程复盘

基于 git log 和会话记录，评估方法论效率。
